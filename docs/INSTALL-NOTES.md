# Install notes & troubleshooting

Environment-specific gotchas worth knowing when installing the framework. None
of these are bugs in the framework — they're host/npm behaviors that surprise
people during setup.

## npm `allow-scripts` policy silently skips postinstall scripts

Some npm setups (notably **npm 11+** with an `allow-scripts` policy in the global
`npmrc`) **block dependency lifecycle/postinstall scripts by default**. The signal
is a *warning*, not an error — the install reports success, so it's easy to miss:

```
npm warn allow-scripts 1 package has install scripts not yet covered by allowScripts:
npm warn allow-scripts   <some-package>@x.y.z (postinstall: node install.js)
```

**Why it matters:** a package whose postinstall downloads a native binary (the
Claude Code CLI, some build tools) can be left **installed but non-functional**
because the binary was never fetched.

**For this framework specifically:** the only affected dependency is `esbuild`,
and its skipped postinstall is **benign** — esbuild ships its prebuilt binary in
the platform package, so the telemetry dashboard still builds. A plain
`npm install` works; no action needed.

**For other global installs on the same machine**, assume a postinstall was
skipped and run it manually:

```bash
# Option A — approve the package's scripts, then reinstall
npm approve-scripts <package>          # (npm approve-scripts --allow-scripts-pending may ENOMATCH for -g pkgs)

# Option B — run the package's own postinstall directly
node "<global-node_modules>/<package>/install.js"   # or install.cjs
```

To check whether your environment enforces this: `npm config get allow-scripts`
(and `allow-scripts-pin`).

## Pointing `oversightDir` at an external content directory

The generated docs and the hand-authored design doc don't have to live in
`~/.claude/oversight`. `oversightDir` is configurable (env `OVERSIGHT_DIR` >
`~/.claude/oversight.json` > autodetect > default), so teams can keep the
**oversight content in its own repo** — separate from this (public) framework —
and point installs at it. `rh-oversight init` prompts for this path on
interactive (TTY) runs, or take `--oversight-dir <path>`.

**Expected layout.** `oversightDir` is the directory that *directly contains* the
design doc. The generators also use a sibling `environment/` dir, and autodetect
looks for the marker `oversight-system/OVERSIGHT_SYSTEM.md`. So the canonical
external layout is:

```
<content-root>/
├── oversight-system/        ← set --oversight-dir to THIS directory
│   ├── OVERSIGHT_SYSTEM.md   ← hand-authored design doc (read)
│   ├── OVERSIGHT_STATE.md    ← generated by `generate-state` (write)
│   └── supervisory-log.md    ← supervisory log (write)
└── environment/
    └── ENVIRONMENT.md        ← generated by `generate-env` (write)
```

The three HTML dashboards (`OVERSIGHT_SYSTEM.html`, `OVERSIGHT_STATE.html`,
`ENVIRONMENT.html`) render to the **workspace root**, not to `oversightDir`.

**Flat-repo caveat.** If your design doc sits at a repo *root* (e.g.
`my-content-repo/OVERSIGHT_SYSTEM.md`) rather than inside an `oversight-system/`
subdirectory, **autodetect will not find it** — the marker it looks for is
specifically `oversight-system/OVERSIGHT_SYSTEM.md`. In that case set the path
explicitly (`--oversight-dir <repo-root>`, or type it at the interactive prompt).
With `oversightDir` = the repo root, `OVERSIGHT_STATE.md` and `supervisory-log.md`
are written into that repo, and `ENVIRONMENT.md` is written to a sibling
`environment/` directory one level up (`<repo-root>/../environment/`). Either
restructure the repo to the canonical layout above, or accept the flat wiring —
both work; the explicit path is what matters.
