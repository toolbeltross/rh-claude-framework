# Document, Data & Project-Tracking Placement

## Principle

Every file written into a project has a category, and every category has a home. The project **root** is for configuration + entry docs only — not a dumping ground. This rule defines the categories and their homes **globally**, and requires **each project to declare its own concrete placement** in its `CLAUDE.md`. It complements `rh-throwaway-artifacts.md` (which covers only disposable `tmp/` artifacts) and `rh-conventions.md` (naming + session-state lifecycle); this rule covers the broader question of *where each kind of file goes*.

## Categories and their homes

| Category | Home | Notes |
|---|---|---|
| **Configuration / entry** | project **root** | `package.json`, `tsconfig.json`, `.gitignore`, `CLAUDE.md`, `README.md`, lockfiles, framework config. Nothing else belongs at root. |
| **Documentation** | `docs/<subdir>/` | Specs/PRDs (`docs/requirements/`), architecture/ADRs (`docs/architecture/`), reference material (`docs/reference/`), guides. Never loose at root. A small number of top-level `docs/*.md` index/overview files are fine if `CLAUDE.md` references them. |
| **Data — committed** | `data/` (or a named domain dir) | Seed data, lookup tables, fixtures intended to live in git. |
| **Data — generated/computed** | `tmp/` or `data/generated/` (gitignored) | Exports, reports, model outputs, caches — anything regenerable. |
| **Temp / throwaway** | `tmp/` (gitignored) | Screenshots → `tmp/screenshots/`; API dumps → `tmp/<subfolder>/`. Defers to `rh-throwaway-artifacts.md`. Never repo root. |
| **Project tracking** | per `rh-conventions.md` | Plans (`PLAN-*.md`), decisions (`DECISIONS.md`), session state (`SESSION_STATE.md`, root, ephemeral, archive when >100 lines), deliberation records (project's chosen dir, e.g. `docs/deliberations/`). |
| **Memory** | **user-scoped — NOT in the repo** | Per-project auto-memory: `~/.claude/projects/<encoded-project-path>/memory/*.md` + its `MEMORY.md` index. Cross-project shared memory: `~/.claude/memory-shared/`. Memory files live under `~/.claude/`, never inside the project working tree — do not commit memory into the repo. |
| **Superseded / historical** | `docs/archive/` | Move here rather than delete when a doc is retired; keep a one-line provenance note. |

## Frontmatter conventions

Placement is *where*; frontmatter is *what metadata the file carries*. Categorized files carry **category-appropriate YAML frontmatter** so they're machine-discoverable:

- **Rules** (`.claude/rules/*.md`): `description:` · `keywords:` · `severity:` (note/warn/block). The `keywords:` field is what the auto-generated Rules Domain Index reads — keep it accurate.
- **Memory** (`~/.claude/.../memory/*.md`): `name:` (kebab-slug) · `description:` (one-line recall hook) · `metadata.type:` (`user`|`feedback`|`project`|`reference`); each is indexed by one line in `MEMORY.md`.
- **Folder manifests** (`_index.md`): YAML frontmatter + a status table (per `rh-conventions.md`).
- **Standing docs / deliberations**: a header block (purpose, source-of-truth, last-updated) — shape is the project's to define.

A project may declare its own frontmatter expectations for its doc/data categories in its File Placement section; this rule's floor is: anything machine-read (rules, memory, manifests) carries the frontmatter its consumer expects.

## Workspace-level vs project-level artifacts (the recurring trap)

Some artifacts are **workspace-scoped** and must NOT land in a project root, even though a tool running with the project as `cwd` may emit them there:

- **Scribe outputs** — `cleanup.md`, `recommendations.md`, `learnings.md` — belong at the **workspace root** (`<workspace>/`), written by the `/rh-quit` scribe. A copy at a project root is a cwd-walkup misfire and is a stale leftover.
- **Oversight dashboards** — `OVERSIGHT_SYSTEM.html`, `OVERSIGHT_STATE.html`, `ENVIRONMENT.html` — are generated by `rh-daily-regen.js` at the **workspace root**. Project-root copies are stale leftovers.

When these appear at a project root: archive them (don't commit), and rely on `.gitignore` to keep them out of git. The canonical, current copies live at the workspace root.

## Per-project instantiation (organic adoption)

This rule is the **global directive**; projects **adopt and extend it organically** — not via a retroactive bulk back-fill. Concretely:

- The global categories above apply everywhere by default, whether or not a project has spelled them out yet.
- A project declares its own "File Placement" (or "Key Directories") section in `CLAUDE.md` **when it's worth it** — when new folders/structure get created, when a placement question actually comes up, or at new-project scaffolding time. Each project **extends** the template for its own layout (monorepo vs flat; `docs/deliberations/` vs `.claude/deliberations/`; whether `data/` is committed; which `docs/<subdir>/` it uses).
- New-project scaffolding **should** include a File Placement section seeded from this rule.
- Where a project has declared its placement, that declaration is authoritative for it; this rule is the floor it must satisfy. Where it hasn't yet, the global categories govern — there is no obligation to back-fill every existing project up front.

`rh-platform-agentbuild/CLAUDE.md` is the reference instantiation to copy from.

## How to apply

1. Before writing a file, classify it (config / doc / data / temp / tracking) and place it in that category's home — never the root unless it's config/entry.
2. If a tool wrote a workspace-scoped artifact (scribe log, oversight HTML) into a project root, treat it as a leftover: archive it; don't commit it.
3. When a project's structure grows (new folders) or a placement question arises, add/extend its `CLAUDE.md` File Placement section then — organically, not as a retroactive sweep across all projects.

## Interaction with adjacent rules

- **`rh-throwaway-artifacts.md`** — owns the disposable-`tmp/` detail; this rule places it in the broader category map and adds docs/data/tracking.
- **`rh-conventions.md`** — owns session-state lifecycle + naming; this rule places those artifacts.
- **`rh-cwd-awareness.md`** — explains the cwd-walkup behavior that causes workspace artifacts to misfire into project roots; this rule says what to do about it.

## Failure mode this rule mitigates

Project roots accumulating stray docs and stale workspace-scoped artifacts (scribe logs, oversight HTML dashboards, orphaned screenshots) with no convention to route or clean them. Evidence: 2026-06-14, `rh-platform-agentbuild` root held 8 such leftovers (3 scribe logs, 3 oversight HTMLs, 2 orphaned PNGs), all stale duplicates of workspace-root canonical copies, with no rule defining where they should have gone.
