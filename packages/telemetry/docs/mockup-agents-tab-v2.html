Review migration 0042_user_schema.sql for safety. Context: we're adding a NOT NULL column to a 50M-row table. Existing rows get a backfill default. Result
The migration is NOT safe under concurrent writes. The backfill runs as a single transaction that holds an ACCESS EXCLUSIVE lock on the table for the entire duration. On a 50M-row table this will take 3-8 minutes, during which all reads and writes to the table are blocked. Recommend: add column as nullable first, backfill in batches of 10K-50K rows with explicit commits, then add NOT NULL constraint as a separate migration.

Performance Considerations

Transcript read frequency for active agents: Parse on each tool event (agent_id-scoped, not every tool event). Reads last ~4KB via readTail (same function we already use on SubagentStop). On a typical agent doing 1 tool/second peak, that's 1 small file read per second per agent — negligible I/O.

Prompt read: One-shot on SubagentStart. Read first line only (~1-5KB). Cache in memory on the active agent record. Never re-read.

Fallback: If transcript is unavailable (race condition, permissions), degrade gracefully to "—" for cost/context, same as current behavior for active agents.
